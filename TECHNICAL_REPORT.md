## Abstract

This report documents the empirical methodology and results of seven systematic benchmarks conducted to guide the architecture of a production-grade Retrieval-Augmented Generation (RAG) system for multilingual question answering over YouTube video content. Each benchmark isolates one architectural variable — transcription model, audio compression, embedding model, language model, retrieval strategy, chunk configuration, and end-to-end Whisper × LLM combinations — and evaluates its impact on a standardised set of metrics. The resulting configuration recommendations are exported as machine-readable JSON and loaded directly by the application at runtime, ensuring that all component choices are traceable to empirical evidence rather than convention.

## 1. Experimental Setup

### 1.1 Benchmark Videos

Three YouTube videos were selected to represent short, medium, and long content durations:

| Label | URL | Duration | Domain |
| --- | --- | --- | --- |
| `short_video` | <https://www.youtube.com/watch?v=l-rBd0F6d9E> | ~5 min | Self-improvement |
| `medium_video` | <https://www.youtube.com/watch?v=ubjW4aq_MAg> | ~17 min | Motivation / Psychology |
| `long_video` | <https://www.youtube.com/watch?v=B5K3Wj6r_HU> | ~42 min | Communication / Law |

All audio was downloaded at 64 kbps m4a via `yt-dlp` with a maximum clip duration of 2700 seconds. Reference transcripts were obtained from YouTube's official caption API via `youtube-transcript-api`.

### 1.2 Computing Environment

All benchmarks were executed on Google Colab with a T4 GPU (CUDA-enabled). Local Whisper models were loaded once per session and cached. Checkpoint serialisation via `pickle` enabled incremental re-execution after interruptions.

### 1.3 Evaluation Questions

Ten factual questions per video were generated by `gpt-4o-mini` from the Whisper API transcript using a structured JSON prompt. Each question includes a `question` string, a `reference` answer grounded in the transcript, and a list of `keywords`. This dataset serves as the evaluation set for Benchmarks 3–7.

### 1.4 LLM-as-Judge

Answer quality is scored on a 1–5 faithfulness scale by `gpt-4o-mini` acting as an evaluator. The judge receives the user question, the AI answer, a reference answer, and the retrieval context. Scoring rubric:

| Score | Interpretation |
| --- | --- |
| 5 | Fully supported — factually correct, covers key points |
| 4 | Mostly correct — minor detail missing or imprecise |
| 3 | Partially correct — topic correct, key facts missing |
| 2 | Addresses topic but facts are wrong |
| 1 | Off-topic, hallucinated, or contradicts context |

## 2. Evaluation Framework

### 2.1 Transcription Quality — WER

Word Error Rate (WER) measures the normalised edit distance between the Whisper output and the YouTube reference caption:

```
WER = (S + D + I) / N
```

where S = substitutions, D = deletions, I = insertions, N = total reference words. Computed via the `jiwer` library after lowercasing both sequences.

### 2.2 Retrieval Quality — MRR and Hit Rate

Retrieval quality is measured by two rank-aware metrics over the top-5 retrieved chunks:

**Mean Reciprocal Rank (MRR@5):**

```
MRR = (1/|Q|) × Σ (1 / rank_i)
```

where `rank_i` is the position of the first relevant chunk for query `i`. A chunk is considered relevant if it contains at least one expected keyword from the evaluation question.

**Hit Rate@5:** The fraction of queries for which at least one relevant chunk appears in the top-5 results.

### 2.3 Retrieval Implementation

Embeddings are computed via `text-embedding-3-small` (1536 dimensions). Similarity search uses cosine distance over pre-computed dense vectors. For Benchmark 6, Reciprocal Rank Fusion (RRF) is evaluated as an alternative to single-scale retrieval.

## 3. Benchmark 1 — Whisper Model Sizes

### 3.1 Objective

Determine the optimal Whisper model variant with respect to transcription accuracy (WER), inference time, and cost across all three video lengths.

### 3.2 Variables

- **Models:** `tiny`, `base`, `small`, `medium`, `large-v3`, `api (whisper-1)`
- **Videos:** `short_video`, `medium_video`, `long_video`
- **Fixed:** 64 kbps audio, reference captions from YouTube

### 3.3 Key Results

| Model | WER (avg) | Cost / run | Inference time |
| --- | --- | --- | --- |
| `tiny` | ~8–12% | $0.00 | fastest |
| `base` | ~5–8% | $0.00 | fast |
| `small` | ~2.4% | ~$0.00097 | moderate |
| `medium` | ~2.2% | ~$0.002 | slow |
| `large-v3` | ~2.1% | ~$0.004 | slowest |
| `api (whisper-1)` | ~2.0% | ~$0.006/min | fastest (API) |

### 3.4 Decision

**`whisper-small`** is selected as the default backend. It achieves a WER of approximately 2.40% — statistically indistinguishable from `medium` and `large-v3` for spoken English content — while requiring no GPU memory beyond a T4 and incurring near-zero cost. The OpenAI API remains available as a premium option for production deployments where GPU availability cannot be guaranteed.

## 4. Benchmark 2 — Audio Bitrate

### 4.1 Objective

Determine the minimum audio bitrate that does not degrade transcription quality, minimising storage and bandwidth requirements.

### 4.2 Variables

- **Bitrates:** 8, 16, 32, 64, 128 kbps
- **Models:** `base`, `small`
- **Fixed:** `medium_video`, YouTube reference captions

### 4.3 Key Results

WER degrades significantly below 32 kbps. At 64 kbps, WER is within measurement noise of 128 kbps across both models. File size at 64 kbps is approximately half that at 128 kbps.

### 4.4 Decision

**64 kbps** is set as the fixed audio download bitrate. This represents the empirically confirmed sweet spot between file size and transcription fidelity.

## 5. Benchmark 3 — Embedding Models

### 5.1 Objective

Select the embedding model that maximises retrieval quality (MRR@5, Hit Rate@5) at minimal cost.

### 5.2 Variables

- **Models:** `text-embedding-3-small` (OpenAI, 1536 dim), `text-embedding-3-large` (OpenAI, 3072 dim), `BAAI/bge-small-en-v1.5` (HuggingFace, 384 dim)
- **Fixed:** `medium_video`, chunk size 500/100, top-5 retrieval, dual Whisper source (API and `small`)

### 5.3 Key Results

`text-embedding-3-small` and `text-embedding-3-large` achieve near-identical MRR and Hit Rate scores. `bge-small-en-v1.5` performs comparably for English content but introduces a dependency on local inference. Retrieval quality is marginally higher on `whisper-small` transcripts compared to API transcripts for this video, suggesting that minor WER differences do not meaningfully impact retrieval.

### 5.4 Decision

**`text-embedding-3-small`** is selected as the default embedding model. It matches the quality of the larger model at approximately 85% lower cost per million tokens ($0.020 vs. $0.130).

## 6. Benchmark 4 — LLM Comparison

### 6.1 Objective

Compare five language models on answer quality (faithfulness score), latency, and cost per query using fixed retrieval context.

### 6.2 Variables

- **Models:** `gpt-5`, `gpt-5-mini`, `gpt-4o-mini`, `hf:qwen2.5-72b`, `hf:llama-3.3`
- **Fixed:** `medium_video`, `text-embedding-3-small`, top-5 chunks, 10 eval questions

### 6.3 Key Results

| Model | Avg Score (1–5) | Avg Latency | Cost / 10 queries |
| --- | --- | --- | --- |
| `gpt-5` | ~5.0 | moderate | highest |
| `gpt-5-mini` | ~5.0 | moderate | moderate |
| `gpt-4o-mini` | ~5.0 | 1.2 s | $0.00154 |
| `hf:qwen2.5-72b` | ~4.8 | slow | $0.00 |
| `hf:llama-3.3` | ~4.7 | fastest | $0.00 |

All models scored 100% accuracy (score ≥ 4) when provided with high-quality retrieval context, confirming that retrieval quality dominates over model capability for factual Q&A tasks.

### 6.4 Decision

**`gpt-4o-mini`** is set as the application default. It achieves maximum faithfulness at the lowest cost among paid models. HuggingFace models (`llama-3.3`, `qwen2.5-72b`) are available as free alternatives for users without OpenAI budget constraints.

## 7. Benchmark 5 — RAG vs. No-RAG

### 7.1 Objective

Quantify the contribution of retrieval augmentation to answer quality on video-specific factual questions.

### 7.2 Design

The same 10 evaluation questions are answered by `gpt-5-mini` under two conditions:

1. **With RAG:** top-5 retrieved chunks prepended to the prompt
2. **Without RAG:** general-knowledge-only prompt, no context

### 7.3 Key Results

| Condition | Avg Faithfulness Score | Accuracy (score ≥ 4) |
| --- | --- | --- |
| With RAG | ~5.0 | ~100% |
| Without RAG | ~2.5–3.0 | 20–40% |

The faithfulness gap is consistent across both `gpt-5-mini` and `gpt-4o-mini`.

### 7.4 Decision

**Verdict:** retrieval is essential for factual video Q&A. Without RAG, large language models either hallucinate plausible-sounding answers or correctly decline to answer, but cannot reliably retrieve specific facts from video content they have not been trained on.

## 8. Benchmark 6 — Chunk-Size Grid

### 8.1 Objective

Determine the optimal chunk size and overlap for transcript segmentation as a function of video duration and Whisper model quality.

### 8.2 Variables

- **Videos:** `short_video`, `medium_video`, `long_video`
- **Whisper models:** `api`, `small`, `base`
- **Chunk configs per video:**
  * Short: (30, 5) to (800, 150) — 11 configurations
  * Medium: (50, 8) to (1500, 250) — 18 configurations
  * Long: (100, 15) to (2500, 400) — 17 configurations
- **Retrieval variant:** single-scale (best config) and multi-scale RRF

### 8.3 Key Results

Optimal chunk size scales monotonically with video duration:

| Video Length | Optimal Chunk Size | Overlap | MRR@5 |
| --- | --- | --- | --- |
| Short (~5 min) | 100–300 tokens | 15–50 | ~0.95 |
| Medium (~17 min) | 400–800 tokens | 75–150 | ~1.00 |
| Long (~42 min) | 800–1500 tokens | 150–250 | ~0.95 |

RRF (multi-scale fusion) consistently matches or marginally exceeds the best single-scale configuration at the cost of higher index complexity. Whisper model choice has a secondary positive effect: higher-quality transcripts yield slightly higher MRR across all chunk sizes.

### 8.4 Decision

A **per-video, per-model chunk recommendation table** is exported as `chunk_recommendations.json`. The QA application loads this file at startup and applies the appropriate chunk configuration automatically at ingestion time based on video duration and the selected Whisper backend. No manual configuration is required.

## 9. Benchmark 7 — Full Whisper × LLM Matrix

### 9.1 Objective

Evaluate every combination of Whisper transcript and LLM answer quality in a single end-to-end experiment to produce a complete performance and cost matrix.

### 9.2 Design

For each of the 6 Whisper transcripts from Benchmark 1:

1. Rechunk at (800, 150) — the empirically optimal config for `medium_video`
2. Embed with `text-embedding-3-small`
3. Retrieve top-5 chunks for all 10 evaluation questions
4. Each of 5 LLMs answers all 10 questions
5. LLM-as-Judge scores all answers

This produces 6 × 5 × 10 = 300 scored answer pairs.

### 9.3 Key Results

Quality is uniformly high (~5.0) across all Whisper × LLM combinations when retrieval quality is adequate. Cost differences are the primary differentiator. The matrix confirms:

- Whisper quality has minimal impact on final answer quality once WER is below ~5%
- LLM cost is the dominant cost factor at query time
- Local Whisper + HuggingFace LLM achieves near-zero marginal cost at comparable quality

### 9.4 Output

The full matrix is exported as `combo_table.csv` and `winners.json`. The application uses this data to populate the benchmark tab's sweet-spot cards and the sortable combination table.

## 11. Summary of Architectural Decisions

| Component | Choice | Evidence |
| --- | --- | --- |
| Audio bitrate | 64 kbps | B2: no WER degradation vs. 128 kbps |
| Whisper default | `local: small` | B1: WER 2.40%, lowest cost/quality ratio |
| Embedding model | `text-embedding-3-small` | B3: matches `large` at 85% lower cost |
| Default LLM | `gpt-4o-mini` | B4: score 5.0, $0.00154/10 queries |
| Chunk size | Adaptive via `chunk_recommendations.json` | B6: no universal optimum; scales with duration |
| Retrieval k | 5 | B7: top-5 sufficient for factual Q&A |
| RAG necessity | Required | B5: accuracy +60–80 percentage points vs. no-RAG |

All decisions are reproducible by re-running `benchmark.ipynb` with the same video URLs and API keys.
